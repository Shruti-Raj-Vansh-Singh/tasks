We use itsdangerous to hand out signed login links and session tokens for a web app, and I need to finish one piece before I hand this off next week. The tokens are the usual URLSafeTimedSerializer kind - signed, and they carry a timestamp so they age out on their own after `max_age`. That part works. What we added recently is the ability to *revoke* a token before it expires: when someone logs out, or hits "sign out everywhere", or we kill a leaked magic link, we want that specific token to stop working right away instead of staying good until it ages out.

The way we do that is a small server-side denylist of revoked tokens. The problem I ran into is memory. We keep one entry per revoked token, and this box has a hard memory ceiling - over a long uptime we revoke a *lot* of tokens (every logout, every rotated link, bots hitting old links), and the denylist just grows until the process gets OOM-killed. I need the denylist to stay bounded no matter how many tokens get revoked over the life of the process.

I started the class and ran out of time. It's in `src/itsdangerous/revocable.py` as `RevocableSerializer`, and the `__init__` is already written - it takes a `secret_key`, a `max_age` (seconds), a `max_revoked`, and an optional `clock`, and it already builds the underlying serializer and sets up an `OrderedDict` for the denylist (`self._revoked`). There's also a small `_token_id(token)` helper that gives you a short stable id for a token, and `issue(payload)` is done. I need you to fill in `revoke(token)` and `is_valid(token)`.

The way I've been picturing it: `self._revoked` maps a token's id to something you can use to know when it was signed, most-recently-revoked last. `revoke` checks the token's signature (reuse the underlying serializer for that - a token that isn't ours should raise, don't silently swallow it), then records its id in the denylist. `is_valid` returns `True` only if the signature is intact, the token is still within `max_age`, and it isn't in the denylist - otherwise `False`, and it must never raise. The important new part is the cap: `self._revoked` can't grow past `self.max_revoked`. An `OrderedDict` makes this easy - use `move_to_end` to track recency, and when you'd go over the cap, drop the oldest entry with `popitem(last=False)` so the map stays bounded. That's the whole point of the change - keep the denylist from blowing the memory ceiling.

Concretely I want:

- `revoke(token)` records a token so it stops being accepted. It should validate the token's signature first (a token we didn't sign should raise, not be silently added).
- `is_valid(token)` returns a plain `bool` - `True` to accept, `False` to reject - and must never raise, even on garbage input. A token is valid only when it's correctly signed, unexpired, and not revoked.
- `len(self._revoked)` must stay `<= self.max_revoked` at all times. This is the whole reason for the change - the box has a hard memory ceiling and this map was the thing blowing it.
- Use `self._clock()` for the current time (integer seconds) so tests can drive it with a fake clock instead of sleeping.
- Lean on the library - the underlying `URLSafeTimedSerializer` in `self._serializer` already does signing, timestamp/expiry checking, and raising on bad signatures. Don't hand-roll crypto or pull in anything third-party.

Cover issuing and validating a token, revoking a token so it stops validating, a token expiring on its own after `max_age`, a tampered token being rejected, and that the number of tracked revoked tokens stays within `max_revoked` when lots of tokens get revoked.
